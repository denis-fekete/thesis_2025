
\chapter{Developing a Bang! application with CVLiBG and Dataset Creator}
\label{bangImplementation}
This chapter demonstrates the use of the proposed CVLiBG mobile library and Dataset Creator application by creating an application for the Bang! card game. 

The purpose of the developed application is not to fully digitize, automate, or provide a computer opponent. Instead, its aim is to provide quick assistance to players by recognizing cards through the smartphone camera and displaying relevant information. Therefore, this chapter serves as a case study showing how the proposed tools can be used together.

\section{Card game Bang!}
Bang! is a western style card game for 4 to 7 players, with a suggested age of 8 or older. The game was created by Italian author Emiliano Sciarra~\cite{bangWiki} and it was released in 2002 by the Italian publisher DV Giochi. In the Czech Republic and Slovak Republic, the game is released and manufactured by Albi. The game is very popular and has achieved 4 million sales worldwide.

Bang! was chosen due to its popularity, but also because most of its cards do not have text-based descriptions. Instead, the game uses symbols. These are not hard to understand,
but it may slow down new or returning players, as they must repeatedly read the rules or ask other players. 

This results in a suitable use case for object detection, as these cards can be quickly scanned before a player's turn or during it, without the need to ask the other players, potentially revealing the card to the other players.


\subsection{Rules and gameplay of Bang!}
The core principle of the game is simple: attack other players, defend and heal yourself, and finally achieve your winning condition before others. 

The Bang! game starts with players receiving cards equal to their health, which is based on role and character they are playing. While the gameplay is the same for all roles, their end goals are different. The game consists of different types of cards: instant effect, long term effect, role, and character cards. The game is turn based, and players can only initiate action during their turn. There are exceptions to this rule, as some cards can be played as a reaction. 

Instant effect cards perform immediate actions, attack other players, take their cards, draw cards from the deck, etc. These cards are usually held by the player and not shown to other players.

Long term effect cards, on the other hand, are placed in front of the player and modify game mechanics such as range.

Players are assigned characters, which are unique and modify how the game is played for each player. For example, \textit{Lucky Duke} allows the player to look at the top two cards and choose one of them instead of drawing at random.

Role cards define the winning condition for the players. Only the Sheriff is known to all players, while the remaining roles are hidden. The game itself doesn’t stop players from lying, deceiving, and forming groups to attack certain players, which, alongside the hidden roles, adds depth and a social element to the game.

\section{Creating object detection model for Bang! card game}
This section demonstrates the use of Dataset Creator for creating a custom fine-tuned object detection model for the Bang! card game. Dataset and models were created in iterative steps, where new image data were annotated, increased in size using synthetic data generation, and then used for training new models.

The resulting model was then integrated into the Android application through CVLiBG.

\subsection{Data gathering and model training}
The first object detection model for the Bang! card game was prepared with a small set of  images captured with the smartphone's camera. To simplify the experiments and serve as a proof-of-concept, the initial dataset contained only two cards. These data were manually annotated in Dataset Creator and exported in the Ultralytics YOLO format.

Synthetic data generation was used to increase the variability of the initial dataset. The generated samples were created from the annotated images and used together with the original data for training directly inside Dataset Creator. 

Additional image data was gradually added in later iterations. Each iteration focused on a limitation observed in the previous model. The first images were captured mostly from a front facing camera angle, which resulted in the model performing worse when cards were viewed  from other angles. Therefore, the next iteration batches included images captured from angled camera positions and under different conditions. 

Previously trained models were then reused for the automatic annotation of new images. The generated annotations still required manual verification, but they reduced the amount of repetitive labeling work.

\section{Android application design}
After training the object detection models with varying inference latency and accuracy, these models were integrated into an Android application using CVLiBG. Application was implemented utilizing CVLiBG's camera handling, object detection, and services as a base.

Game specific parts of the application, such as card descriptions, settings, or navigation, are not part of CVLiBG and are specific to this application. This separation should demonstrate the capabilities of CVLiBG, as it provides object detection infrastructure, while the final application utilizes its features in the context of the Bang! card game.

The application was set up within the chosen IDE, and a new blank Android project was created. CVLiBG was integrated by adding it to the project's dependencies inside the \texttt{build.gradle.kts} file. (see Appendix~\ref{mobileInstall} for the integration of CVLiBG into Android projects).


\subsection{Activities, fragments and navigation}
This application utilizes a single Android \texttt{AppCompatActivity} hosting multiple fragments, each providing different functionality. This is achieved using \texttt{NavHostFragment}\footnote{NavHostFragment: \url{https://developer.android.com/reference/androidx/navigation/fragment/NavHostFragment}}, which allows switching between multiple fragments. This approach also allows sharing a single implementation of bottom navigation buttons.

Navigation host fragment utilizes \texttt{NavigationGraph}, an \texttt{.xml} file defining fragments and transitions between them. 

The application is mostly made up of three main fragments, which the navigation host fragment switches between: camera, settings, and search fragments. All fragments share a single activity layout, which is set up in such a way that the navigation buttons at the bottom are transparent, and the current fragment fills the whole screen.

Furthermore, additional fragments, such as inventory, role, and character, describing the current game state are placed on top of the main activity's fragment container. This allows all other fragments to see and use the functionality of these fragments.

Lastly, as the main activity will use the camera, it is necessary to request permission for using it, a helper method \texttt{CommonUtils.checkCameraPermission()} was used. Likewise, these permissions must be added to the \texttt{AndroidManifest.xml}.

\subsection{Services and detector registry}
This application uses CVLiBG's services for loading asset files containing game specific information. Instead of coding each card into the application, a set of \texttt{.json} files is used. These files define each card's title, description, image, etc.

\texttt{CardDetailsService} is derived from the template \texttt{AssetService} and its data class \texttt{CardDetail}. These data objects contain specific information for the given card. Loaded files are located in the \texttt{details/} directory under the project's asset folder. For the convenience of manually editing these files, cards were separated into different files based on their type (weapons, characters, etc.)

\texttt{Class2LinkService} also uses the \texttt{AssetService} and employs a simple data class, \texttt{Class2Link}. The purpose of this service is to translate class integer indexes from the results of the detector into human readable string ids. These ids are used instead of integer values for the \texttt{CardDetail}. This allows for a more readable and manageable system. Card details also contain links to related cards. Without this "translation layer", it would be necessary to remember which integer value belongs to which card. Furthermore, if the integers changed, for example, by adding a new class, the card details would have to be rewritten. 

\texttt{SettingsService} uses the template \texttt{ConfigService} and \texttt{UserPreferences} as its data class. Its purpose is to provide common application settings, such as language and font size. Furthermore, this application allows users to choose which detector should be used, as some hardware configurations might benefit more from Android's Android's Neural Networks API library. This option is also paired with the \texttt{DetectorRegistry} to ensure that incompatible models and detectors will not be used.

\texttt{GameStateService} utilizes the template \texttt{StorageService} and its \texttt{GameState} data class. This service is mostly used to store the current game state, such as the inventory, role, and character used by the player.

\subsection{Inventory and shared game state }
To further help players, an option to add cards to the "inventory" was added. This allows players to store cards they "scanned" or searched for. Action cards and long effect cards, such as weapons, can be added to the inventory. Whereas roles and characters can be set so that players can quickly look them up without the need to search for them in the inventory.

However, actions such as adding a card to the inventory or changing the current character require communication between fragments. For example, the fragment showing details of the card can add a card to the inventory, and the inventory fragment must react to these changes. 

This was achieved using Android's \texttt{ViewModels}\footnote{: \url{https://developer.android.com/topic/libraries/architecture/viewmodel}} and \texttt{StateFlow}\footnote{StateFlow: \url{https://developer.android.com/kotlin/flow/stateflow-and-sharedflow}}. View models allow a fragment's data to survive rotations and other view destroying events. Likewise, the \texttt{activityViewModel} provides the same functionalities and allows other fragments to reuse and share the same model, essentially providing a way to communicate with each other as long as they are under the same activity. This view model contains \texttt{StateFlow} with the current inventory, character, role, or flags for hiding and showing inventory. 

Inventory fragment reacts to these changes using the \texttt{collect()} method. This ensures that changes made in a different fragment or on a different thread will be correctly reacted to by the fragments.

% =============================
\subsection{Camera fragment}
The fragment hosting the camera is the main point of this application. It hosts the camera preview, \texttt{BangDetectionOverlay}, and metrics overlay. It is implemented by the \texttt{CameraFragment}.

The camera control is fully managed by the \texttt{CameraController}. The controller is placed inside the \texttt{ViewModel} to preserve its state, and only the \texttt{start()} method is called on each view creation. This allows the \texttt{CameraController} to remain unchanged when the view is destroyed or recreated. Likewise, the image analyzer is set up in a similar manner.

The results of the analysis are collected and passed to the \texttt{BangDetectionOverlay}, a subclass of the \texttt{DetectionOverlay} for this specific game.

Furthermore, fragment supports both realtime and detailed analysis modes. By default, a realtime analysis is performed on each frame, allowing users to click onto detections, which will bring them to the details of each card. 

Switching into detailed mode will keep the camera preview active until the results arrive. After new results with detailed analysis are collected by the fragment, the detection overlay will display these results, and the camera controller will stop.

This is not part of the CVLiBG, as some developers might want to implement this functionality differently and switch to another fragment. To allow such behavior, it is necessary to stop the camera controller manually, as in this case.

Detailed mode also allows clicking on the detected card, but it also enables the inventory from which cards can be used.

\subsection{Card details}
\label{DetailsFragment}
Card details is a screen, or fragment presented to the user after interacting with the detected card. This screen contains information about the card. Small games, such as Bang!, can implement each detail card as a separate custom screen. However, some games can contain hundreds of different items or cards, and creating a custom screen for each of these items would be unreasonable.

\texttt{CardDetailsFragment} solves this by utilizing services and dynamically loading details for the given card. These values are loaded from the \texttt{.json} files and are loaded as separate card objects, which can be accessed later by the \texttt{CardDetailsService}. This approach uses a single fragment and fills it with details from the loaded JSON object.

\subsection{Card search}
Card recognition through the camera is the primary interaction method of the application. However, the application also provides a manual card search. This screen serves as a fallback mechanism in situations where the card cannot be detected reliably, for example due to poor lighting.

The search functionality also aims to reduce disruption during gameplay. If another player plays a card and the user wants to read, the card can be searched manually without asking the other player to show the card again by placing the smartphone on top of the card. 

Search was implemented using Android's \texttt{RecyclerView}\footnote{RecyclerView: \url{https://developer.android.com/develop/ui/views/layout/recyclerview}} and \texttt{SearchView}\footnote{SearchView: \url{https://developer.android.com/reference/android/widget/SearchView}}, which allowed simple integration by loading images from the \texttt{CardDetailsService}.

\subsection{Settings and Benchmark}
The last fragment consists of UI elements for interacting with the \texttt{settingsService}. This class uses the template \texttt{ConfigService} with its data class \texttt{UserPreferences}. This service is used to store and load values, such as the application's language, selected detector, font size, etc.

Settings also contain a link to the \texttt{BenchmarkFragment}. This fragment utilizes CVLiBG's \texttt{PerformanceBenchmark} for testing inference latency of different detectors.
