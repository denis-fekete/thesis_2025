
\chapter{Developing a Bang! application with CVLiBG and Dataset Creator}
\label{bangImplementation}
This chapter starts with basic information about the card game Bang!, followed by creating a first model and an example use of the dataset tool. Next, the chapter continues with the implementation of a mobile application at a higher abstraction level, explaining the design choices.

\section{Card game Bang!}
Bang! is a western style card game for 4 to 7 players, with a suggested age of 8 or older. The game was created by Italian author Emiliano Sciarra \cite{bangWiki}, it was released in 2002 by the Italian publisher DV Giochi. In the Czech Republic and Slovak Republic, the game is released and manufactured by Albi. The game is very popular and has achieved 4 million sales worldwide.

\subsection{Rules of Bang!}
At the start of the game, players are randomly assigned one of the roles: Sheriff, Sheriff's Assistant, Bandit, or Renegade. While the game-play is the same for all roles, their end goals are different. \textbf{Sheriff's} role is to get rid of the renegade and all the bandits. The sheriff also gets bonus health; however, everybody knows who the sheriff is, while all other players keep their roles a secret. \textbf{Assistants}, unlike the sheriff, do not receive any additional bonuses, and their goal is to protect the sheriff. \textbf{Bandits}, on the other hand, are targeting all other players. Their win condition is to get rid of the Sheriff. \textbf{Renegade} is the last role and is similar to bandits. However, there can only be one Renegade, and the winning condition of this role is to win the game by getting rid of the Sheriff while the Renegade and Sheriff are the only players left in the game.

Except for the roles, players are also assigned characters. These characters modify how players play the game. For example, \textit{Lucky Duke} allows the player to look at the top two cards and choose one of them instead of drawing at random. \textit{Willy the Kid} removes the limit on how many damaging cards Bang! can be played in one round by the player. And the list goes on.

The core principle of the game is simple: attack other players, defend and heal yourself, and finally stop other players from achieving their goals while trying to achieve yours. The game can be relatively explosive, and players can be taken by surprise if other players group together and start attacking them. The game itself doesn’t stop players from lying, deceiving, and forming groups to attack certain players, which, from personal experience, makes it even trickier and more fun to play.

\section{Creating object detection model for Bang! card game}
To test the capabilities and usability of the proposed Dataset Creator, this tool was used to create an annotated dataset and train object detection models for the Bang! card game.

\subsection{Data gathering and annotating}
In order to create an object detection model, it is necessary to gather image data with the game cards. These can be found online or taken with a smartphone's camera. Manually annotating a few dozen images might not take too much time; however, once developers start annotating hundreds of images, the task becomes more burdensome. 

To help developers with this problem, the Dataset Creator provides an automatic annotation functionality. This, however, requires a working object detection model capable of recognizing and detecting game objects. The recommended approach is to annotate a few examples of each game object, use synthetic data generation, and utilize this data to train the first model. This model can later be used for annotation, and while the detections might not always be precise, it should speed up the further annotation of image data. 

However, it should be noted that iterating and creating newer models after annotating only a few image data might not be worth the time spent on training. Developers should find a balance based on the amount of time it takes for their machine to train one model.

\subsection{Training the models}
Before training, it is required to export the dataset. Choosing the correct settings for training is mostly a matter of trial and error, especially since the batch size and number of workers will be limited by the hardware's capabilities of the computer used for the training. Exported datasets are in Ultralytics YOLO model format. These saved datasets can be used for training outside of the Dataset Creator , even by other training solutions; however, conversion might be required. The \textbf{Dataset Creator} is primarily meant for hobbyists and small developers who use their machines for training as well.

\section{Implementing mobile application}
After creating a new project in the integrated development environment of the developer's choice, the library needs to be added to the project's dependencies (see \ref{mobileInstall}).

The starting point of the application is \texttt{MyApp}, which is derived from the \texttt{Application} class. This is where \texttt{DetectorRegistry} entries are defined to prevent mismatched model and \texttt{Detector} versions (see \ref{DetectorRegistry}), as these will be changeable in the settings menu.

The next part is an Android's \texttt{Activity} hosting all the functionalities of the application. Instead of using multiple activities, this application uses only one activity and changes fragments. As this activity will contain use cases and functionality that require the camera, it is necessary to request permission to use it; this can be achieved with \texttt{CommonUtils.checkCameraPermission()} method.

Furthermore, it is necessary to change the \texttt{AndroidManifest.xml} file and add the permission request to use the camera (see \ref{androidManifestExample} for an example of the \texttt{AndroidManifest.xml}).

Navigation is achieved using \texttt{NavHostFragment} \footnote{Android API -- Navigation host fragment:\url{https://developer.android.com/reference/androidx/navigation/fragment/NavHostFragment}} and \texttt{NavigationGraph}. This approach changes the fragments displayed as content while staying inside the same activity. Furthermore, to simplify loading details for each card, a link system was set up.

\section{Fragments}
Fragments are parts of the UI. By design, these can be swapped without changing the activity, allowing for the reuse of the navigation functionality and logic from the main activity. 

All fragments in this application are derived from \texttt{BaseFragment} (not part of the CVLiBG). This simple class is derived from \texttt{Fragment} and implements automatically generated view binding for each layout, resulting in less error prone and cleaner code \footnote{Android API -- View binding: \url{https://developer.android.com/topic/libraries/data-binding/generated-binding}}.

% ===========================================================================================

\subsection{Life-cycle scope of fragments}
This application mostly uses \texttt{Fragments}. These have separated fragment lifecycle and view lifecycle. Meaning that while the fragment is kept in memory, its view might be destroyed; such situation occurs when minimizing the application \cite{fragmentLifecycle}.

\subsection{Camera Fragment}
The \texttt{CameraFragment} is responsible for creating \texttt{CameraController}, providing \texttt{Detectors} to \texttt{ImageAnalyzer}, and connecting its outputs to the UI elements.
This fragment is responsible for setting up the \texttt{CameraController}, initializing detectors, and adding on-click behavior for user interaction with detected objects. The abstract \texttt{DetectionOverlay's} \texttt{onDetectionClicked} is a callback function invoked on user interaction with detected objects. The lambda function uses the \texttt{Class2LinkService} to convert the integer class value into a string link Id used by the navigation controller to open the details card.

\begin{verbatim}      
    detectionOverlay.onDetectionClicked = { detection ->
        val bundle = bundleOf("id" to class2linkService
            .items[detection.classIndex]!!.linkId)
        findNavController().navigate(R.id.cardDetailsFragment, bundle)
    } 
\end{verbatim}

The \texttt{BangDetectionOverlay} provides a reference to the clicked detection object. This application implements a link system, which allows loading details of each card from a JSON file. These files are a list of JSON objects containing values such as title, image, description, and links to other cards. However, defining these links by the class ID would be tiresome and hard to read for most developers. 

Because of that, a \texttt{Class2LinkService}, is used to convert \textbf{integer} class values, which are the output retrieved from the \texttt{BangDetectionOverlay}, into \textbf{string-based} link identifiers used for navigation when displaying details for each card or symbol. The service is based on the template \texttt{AssetService} and uses the \texttt{Class2Link} data class.

Further, it is required to provide \texttt{Detector} classes for the \texttt{CameraController}. This application uses the \texttt{SettingsService} for storing the currently used detectors, which allows users to change detectors and models in settings.

\begin{verbatim}
    private fun setupDetectors() {
        realtimeDetector = DetectorRegistry.createDetector(
            settingsService.data.realtimeModel)
        qualityDetector = DetectorRegistry.createDetector(
            settingsService.data.precisionModel)
    }
\end{verbatim}

\label{BangDetectionOverlay}
This fragment also contains the \texttt{PreviewView}, \texttt{MetricsOverlay} and \texttt{DetectionOverlay} derived class, \texttt{BangDetectionOverlay}. The layout follows the recommended approach with \texttt{PreviewView}, \texttt{BangDetectionOverlay}, and \texttt{MetricsOverlay} stacked on top of each other (see \ref{CameraFragmentOverlays}).

The \texttt{BangDetectionOverlay} defines a \texttt{drawDetections()} method, which contains code for drawing detections onto the screen. It iterates through all detection objects and uses \texttt{the scaleDetectionToScreenRect()} method from the \texttt{DetectionOverlay} to scale detections, as the camera resolution may not match the screen's resolution.

\subsection{DetailsFragment}
\label{DetailsFragment}
Details fragment loads values from the loaded JSON files and displays its contents. This is achieved with \texttt{CardDetailsService}, which uses the template \texttt{AssetService} and the \texttt{CardDetail} data class. Each object contains an \texttt{id} element, which is used as a key for the service. This link system also allows defining other IDs in the list of string ids, the \texttt{links} field. These links are translated as clickable objects, moving the user to another details card.

\subsection{SettingsFragment}
The last fragment consists of UI elements for interacting with the \texttt{settingsService}. This class uses the template \texttt{ConfigService} with its data class \texttt{UserPreferences}. This service is used to store and load values, such as the application's language, selected detector, font size, etc.
